## Соглашения по коду (обязательно к соблюдению)

### Python / Django
- **Стиль:** строго PEP 8. Максимальная длина строки — 88 символов (black-compatible).
- **Именование:** snake_case для переменных/функций, PascalCase для классов, UPPER_CASE для констант.
- **Типизация:** использовать аннотации типов для публичных функций и методов классов.
- **Комментарии:** только там, где логика неочевидна. Не дублировать то, что ясно из кода.
- **Импорты:** порядок — stdlib → third-party → local (соответствует isort).
- **Тесты:** pytest + pytest-django. Не использовать mock для БД без крайней необходимости.

### Запрещено
- Хранить секреты (SECRET_KEY, API-ключи) в коде — только через `.env` + `python-dotenv`.
- Делать коммиты без явного запроса пользователя.
- Менять архитектурные решения без согласования.

### Консольный вывод (обязательно)
- **Только ASCII** в любых `print()`/`stdout.write()` в скриптах (`ml_analytics/`, management-команды).
- Запрещены: символы вне диапазона 0x00–0x7F (псевдографика, «─», «█», «→», кириллица в выводе).
- Разрешённые разделители: `-`, `+`, `|`, `=`, `*`.
- Причина: Windows-терминал с cp1251 не отображает Unicode вне Latin-1 — текст превращается в ромбики.

## Соглашения по тексту ВКР

- **Стиль:** строгий академический. Без разговорных оборотов, без «воды».
- **Лицо:** третье лицо («в работе рассматривается», «система реализует», «автором предложен»).
- **Терминология:** единая по всему тексту (LMS, RAG, ChromaDB — без замены синонимами).
- **Структура абзаца:** тезис → аргументация → вывод.
- **Источники:** оформлять по ГОСТ Р 7.0.5-2008.
- **Формулы:** LaTeX-нотация, нумерация по главам (1.1), (1.2) и т.д.
